Pulse Check

“RAG 死了”的三种声音

  • 检索命中率不稳定，内容不相关导致体验崩塌。
  • 长上下文成本激增，效果仍难超过针对性微调。
  • 评估体系缺失，无法向业务方证明提升了什么。
共识：朴素 RAG 模式在复杂场景中难以满足可控、可追溯、可优化的要求。

信任

数据资产

LLM Reality

大模型的结构性约束仍然存在

  • 知识截止：参数化知识无法覆盖最新、专有、实时信息。
  • 幻觉风险：概率生成天性让模型在缺乏证据时编造答案。
  • 领域深度：通用语料训练的模型对企业内部语境理解有限。
  • 可验证性：缺乏来源引用，审计链路缺失。
引用 context 的 1.1 节内容，强调没有外部上下文补强，大模型在企业级场景仍不够用。 铺垫 RAG 作为“现实锚点”的必要性。
Value Lens

RAG 仍是企业级 AI 的基础设施

准确性 & 可信任

让回答锚定真实文档，显著降低幻觉，提供可审计的引用链路。

实时性 & 专有性

即时接入最新公告、票据、运营数据，实现 just-in-time 知识。

成本效率

更新知识库远比重新训练/微调模型更廉价，部署风险更小。

个性化扩展

通过多库路由适配不同业务线，实现高度定制化的智能助手。

概括 1.3 节的价值主张，说明 RAG 依旧是企业级智能的底座。 强调对话仍然聚焦 ROI，而不是情绪化的“死”。

从 RAG 到上下文工程的范式转移

我们不再只关注一次性“检索 + 生成”，而是构建一条可观测、可编排的上下文供应链。

  • 数据 → 检索 → 推理 → 运维的闭环设计。
  • 关注点从“提示词技巧”转为“上下文治理”。
  • 让知识库、工具、代理协同演化，持续提升质量。
总结 context 第七部分提出的范式转移概念，呼应“上下文工程”。 作为后续各个模块的过渡。

Retrieval Core

第二步：检索心脏 = 嵌入模型 × 向量数据库

  • MTEB 只是起点，必须结合自有数据集离线评估，关注召回率与延迟。
  • 维度、窗口、推理延迟在准确性与成本之间权衡；API 与自部署决定合规边界。
  • Milvus/HNSW/IVF 等索引组合，实现高吞吐低延迟的近似最近邻搜索。
  • 结合元数据过滤与多路召回，提升复杂问答一致性。

Quality Loop

第三步：用重排与评估闭环守住体验

多阶段检索

BM25 + 语义检索组合，借助重排模型（如 Cohere ReRank）提高精准度。

自动化评估

围绕真实性、相关性、完整性构建 LLM 评估 + 人审混合体系，记录指标趋势。

可观测性

采集检索命中率、响应延迟、成本，触发回归测试与知识库更新。

Beyond Naive

演进路线：自适应、纠正型与代理式 RAG

Self-RAG

模型自我反思词元，按需检索并评估证据支撑度，提升事实性。

CRAG

为检索结果打分，低质量时触发网络搜索等补救路径，保证召回覆盖。

Agentic RAG

引入代理规划与工具调用，拆解多跳任务，实现可编排的知识工作流。

Dify Blueprint

Dify 实战：从知识库到 Agent 的编排

  1. 知识库存储：上传政策、手册、FAQ，自动处理分块与向量化。
  2. 画布编排：拖拽 Start → Retrieval → LLM → Answer，快速上线朴素 RAG。
  3. 质量分支：通过评分节点 + Conditional，实现 CRAG 式兜底。
  4. 代理化：引入 Agent 节点，赋予工具箱，完成多跳推理与外部搜索。
